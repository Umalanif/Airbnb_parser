## РЕЗЮМЕ ПРОЕКТА:

**Задача:** Разработка изолированного воркера `jobs/parser.js` для планировщика `Bree`, выполняющего парсинг цен с закрытого API Airbnb (StaysPdpSections) с использованием персистентных запросов (GraphQL).
**Стек технологий:** Node.js (ESM), `got-scraping` (HTTP-клиент), `zod` (валидация схем), `@prisma/client` (ORM), `pino` (логирование).
**Архитектурная специфика:** Отсутствие глобального состояния, инстанцирование Prisma внутри цикла выполнения, строгая работа с хидерами сессии и `URLSearchParams`, graceful shutdown через `parentPort`.

## ПЛАН РАЗРАБОТКИ:

- [ ] **Шаг 1: Конфигурация логгера и схем валидации (Zod)**
  - **Тип:** Изолированный компонент
  - **Требования:** Инициализировать инстанс `pino` для логирования. Создать строгую схему `zod` для безопасного парсинга ответа Airbnb: схема должна валидировать наличие пути `data.presentation.stayProductDetailPage.sections.sections` (массив) и описывать структуру элемента массива, где `sectionId === "BOOK_IT_SIDEBAR"`, для последующего безопасного извлечения данных о цене.
  - **Включенные компоненты:** Инстанс логгера `pino`, схема ответа `zod`.

- [ ] **Шаг 2: Утилита конструирования URL и параметров запроса**
  - **Тип:** Изолированный компонент
  - **Требования:** Написать чистую функцию для формирования целевого URL. Функция должна принимать `listingId`, `checkIn`, `checkOut` и другие параметры, кодировать `listingId` в Base64 (в формате `StayListing:{id}`), формировать объекты `variables` и `extensions`, сериализовать их в JSON и использовать `URL` и `URLSearchParams` для безопасного добавления к базовому URL `StaysPdpSections` вместе с `operationName`, `locale` и `currency`.
  - **Включенные компоненты:** Функция генерации URL, утилита Base64-кодирования.

- [ ] **Шаг 3: [Smoke Test - Логика подготовки данных]**
  - **Критерий успеха:** Схемы `zod` успешно компилируются; функция генерации URL возвращает валидную строку с корректно заэнкоженными JSON-строками в параметрах `variables` и `extensions`.
  - **Действие:** Если работает — переход к следующему шагу без рефакторинга.

- [ ] **Шаг 4: Интеграция БД и выполнение HTTP-запроса**
  - **Тип:** Связанная группа
  - **Требования:** Реализовать логику внутри асинхронной функции: инициализация `PrismaClient`, запрос записи сессии (`id === 'airbnb_main'`). В случае отсутствия сессии — логирование ошибки в `pino` и прерывание. При наличии сессии: выполнение GET-запроса через `gotScraping.get()`. В запрос должны быть строго переданы хидеры из БД (`cookie`, `x-airbnb-api-key`, `user-agent`) и хардкод-хидеры `sec-ch-ua` (Chromium v146), `sec-ch-ua-platform`, `x-airbnb-graphql-platform`.
  - **Включенные компоненты:** Чтение из `Session` (Prisma), клиент `got-scraping`, формирование хидеров.

- [ ] **Шаг 5: Валидация ответа и персистентность данных**
  - **Тип:** Связанная группа
  - **Требования:** Пропустить тело ответа `gotScraping` через метод `schema.safeParse()`. При ошибке валидации — залогировать `.error.format()` в `pino` и прервать выполнение текущей итерации без выброса исключения (non-fatal error). При успехе — извлечь цену, нормализовать её (конвертация micros в float) и сохранить запись в БД с помощью `prisma.priceLog.create`.
  - **Включенные компоненты:** Обработка результата `zod`, нормализация данных, запись в `PriceLog` (Prisma).

- [ ] **Шаг 6: [Smoke Test - Парсинг и запись]**
  - **Критерий успеха:** При мокировании сессии и ответа от API, скрипт успешно валидирует данные и вызывает метод создания записи Prisma (без необработанных исключений).
  - **Действие:** Если работает — переход к следующему шагу без рефакторинга.

- [ ] **Шаг 7: Обертка жизненного цикла воркера Bree**
  - **Тип:** Связанная группа
  - **Требования:** Обернуть всю логику в главную точку входа. Добавить обработчик событий Node.js `parentPort.on('message')`. При получении сообщения `'cancel'`, скрипт должен вызвать `prisma.$disconnect()` и `process.exit(0)`. Добавить блок `finally` в основное тело выполнения для гарантированного вызова `prisma.$disconnect()` после завершения цикла парсинга.
  - **Включенные компоненты:** Управление процессом (`parentPort`, `process.exit`), управление ресурсами БД (graceful disconnect).

- [ ] **Шаг 8: [Smoke Test - Жизненный цикл воркера]**
  - **Критерий успеха:** Воркер корректно перехватывает сообщение `'cancel'`, закрывает соединение с БД и завершает процесс с кодом 0; при штатном завершении также вызывается `$disconnect()`.
  - **Действие:** Если работает — план завершен, код готов к выгрузке (вывод ИСКЛЮЧИТЕЛЬНО валидного ESM JavaScript кода для `jobs/parser.js`).

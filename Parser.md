РЕЗЮМЕ ПРОЕКТА:
Задача: Разработка высоконагруженной, автономной системы мониторинга цен и доступности целевых объектов (Listings) с функционалом Data Extraction, Delta Detection и Alerting.
Стек технологий: Node.js (ES Modules, Pure JS), Bree.js (оркестрация), Crawlee (CheerioCrawler / PlaywrightCrawler), Prisma + SQLite (с WAL модом), Zod (валидация), Pino (логирование), Telegram Bot API + Bottleneck (уведомления).
Архитектурная специфика: Event-Driven Orchestrator-Worker модель. Главный поток управляет расписанием и Graceful Shutdown, изолированные воркеры выполняют задачи парсинга. Строгая идемпотентность, встроенная защита от банов (обработка 403, WAF) и лимитов API Telegram.

ПЛАН РАЗРАБОТКИ:
[ ] Шаг 1: Обновление схемы базы данных (Data Model)

Тип: Изолированный компонент

Требования: Модифицировать prisma/schema.prisma с учетом новых требований ТЗ. В модель Listing добавить уникальное поле url (или sku) и title. В модель PriceLog добавить поля isAvailable (Boolean), delta (Float) и переименовать/настроить поле timestamp в capturedAt. Убедиться, что связи между таблицами сохранены.

Включенные компоненты: schema.prisma, конфигурация SQLite (journal_mode=WAL, connection_limit=1).

[ ] Шаг 2: [Smoke Test - Миграция БД]

Критерий успеха: Команды npx prisma format, npx prisma generate и npx prisma db push выполняются без ошибок. Структура SQLite обновлена.

Действие: Если работает — переход к следующему шагу без рефакторинга.

[ ] Шаг 3: Разработка Notification Gateway (Telegram + Bottleneck)

Тип: Изолированный компонент

Требования: Создать сервисный модуль для маршрутизации уведомлений через Telegram Bot API. Обернуть методы отправки в bottleneck для предотвращения ошибки 429 (Rate Limit). Реализовать два абстрактных метода: sendBusinessAlert(payload) (для цен и статусов) и sendSystemAlert(payload) (для 403 ошибок, падений воркеров). Интегрировать Pino для логирования статусов отправки.

Включенные компоненты: Инстанс bottleneck, HTTP-клиент (встроенный fetch или got-scraping) для Telegram API, утилита форматирования сообщений.

[ ] Шаг 4: [Smoke Test - Notification Gateway]

Критерий успеха: Вызов тестового скрипта отправляет 5 сообщений подряд; bottleneck успешно ставит их в очередь, сообщения приходят в Telegram без блокировок API.

Действие: Если работает — переход к следующему шагу без рефакторинга.

[ ] Шаг 5: Инициализация Orchestrator (Bree.js Control Plane)

Тип: Связанная группа

Требования: Создать главный файл приложения (например, index.js). Инициализировать инстанс Bree с базовой конфигурацией расписания (Cron) и ограничением maxWorkers. Настроить перехват сигналов SIGINT и SIGTERM. При получении сигнала оркестратор должен корректно рассылать сообщение cancel всем активным воркерам и дожидаться их завершения через bree.stop() (Graceful Shutdown).

Включенные компоненты: Точка входа приложения, конфигурация Bree.js, обработчики событий процесса Node.js.

[ ] Шаг 6: [Smoke Test - Жизненный цикл Orchestrator]

Критерий успеха: При запуске index.js Bree успешно стартует фиктивный (dummy) воркер. При нажатии Ctrl+C (SIGINT) оркестратор отправляет сигнал отмены, дожидается остановки воркера и завершает процесс с кодом 0.

Действие: Если работает — переход к следующему шагу без рефакторинга.

[ ] Шаг 7: Создание Crawler Worker (Data Extraction Engine)

Тип: Связанная группа

Требования: Разработать воркер (Execution Node) с использованием Crawlee (CheerioCrawler). Логика: инициализация изолированных PrismaClient и Pino, выборка всех Listing, где isActive === true. Заполнение RequestQueue URL-адресами объектов. В requestHandler реализовать логику извлечения текущей цены и статуса доступности (Sold Out / Unavailable). Добавить Anti-Bot логику: счетчик последовательных ошибок 403, при превышении порога вызывать await crawler.teardown() и логировать системный алерт.

Включенные компоненты: Инициализация Crawlee, RequestQueue, парсинг DOM, обработчик ошибок (429 Backoff, 403 Teardown).

[ ] Шаг 8: [Smoke Test - Извлечение данных Crawlee]

Критерий успеха: Воркер успешно вытаскивает целевые URL из БД, проходит по очереди, извлекает цену/статус (или корректно симулирует 403) и не имеет глобального разделяемого состояния.

Действие: Если работает — переход к следующему шагу без рефакторинга.

[ ] Шаг 9: Реализация Delta Detection и персистентности (Data Gateway)

Тип: Связанная группа

Требования: Встроить логику вычисления дельты внутрь requestHandler воркера. Перед записью запрашивать последнюю известную цену для объекта. Вычислять delta = currentPrice - previousPrice. Пропускать извлеченные данные через схему Zod (safeParse). Записывать новые данные в БД (использовать prisma.priceLog.create или upsert для гарантии идемпотентности). Если delta !== 0 или изменился статус доступности, асинхронно вызывать метод sendBusinessAlert из Notification Gateway.

Включенные компоненты: Схемы валидации Zod, бизнес-логика сравнения (Delta), транзакции Prisma, интеграция с Notification Gateway.

[ ] Шаг 10: [Smoke Test - End-to-End Pipeline]

Критерий успеха: При наличии тестового объекта в БД воркер запускается, парсит актуальную цену, сравнивает с предыдущей, записывает корректный лог в SQLite с заполненным полем delta, и (при изменении цены/статуса) доставляет оповещение в Telegram. При штатном завершении очереди или получении сигнала cancel вызывается prisma.$disconnect().

Действие: Если работает — план завершен, архитектура полностью реализована.
